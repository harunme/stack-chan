# ｽﾀｯｸﾁｬﾝ開発ロードマップ

最終更新日: 2026-03-27

## 目的

この文書は、ｽﾀｯｸﾁｬﾝの実行指向のロードマップを定義するものです。

中心となる目標は、現在の ｽﾀｯｸﾁｬﾝ の価値を損なわずにセットアップや開発の負担を減らすことです。

- 高速に反復できること
- Hackable であること
- ファームウェア、MOD、AI、ツーリングの各層で拡張可能であること

## 計画原則

- Browser-first だが browser-only ではない。高度なローカル開発は引き続き可能としつつ、一般的なワークフローでは Moddable や ESP-IDF のセットアップを前提にしない。
- 機能拡張より先にインターフェースを標準化する。複数ベンダー対応や複数エージェント対応を進める前に、mod 形式、AI パイプライン、ログ、イベント契約を安定させる。
- Host/MOD アーキテクチャを維持する。既存の高速な MOD 反復サイクルは戦略的な強みであり、引き続き中核に置く。
- 高度な AI 機能を増やす前に AI の挙動を安定化する。TTS の途切れやランタイム挙動の不透明さは、Vision、Wake Word、Swarming より先に解決する。
- 初学者向けと上級者向けのワークフローを分断しない。Visual ツールや教育向けフローも、上級者向け mod 開発と同じランタイムとパッケージングモデルに収束させる。

## 現在のベースライン

2026-03-27 時点で、このリポジトリにはすでにいくつかの強い土台があります。

- Host と MOD が分離されており、MOD 単体の高速デプロイが可能
- ブラウザベースのファームウェア書き込み機能がすでに存在する
- BLE 経由のブラウザ設定変更機能がすでに存在する
- TTS 対応と AI 関連デモは存在するが、セットアップは依然として外部サービスと手動設定に依存している
- ローカル開発環境のセットアップは依然として重く、特に Moddable 環境に慣れていない利用者には負担が大きい

したがって、このロードマップでは既存の要素を置き換えるのではなく、ひとつの一貫したプラットフォームとして統合していくことを重視します。

## 戦略優先事項

1. 今後 5 年を支えるために 2026 年中に必要な最小限のアーキテクチャリファクタを定義する。
2. 標準的な開発ループを、より多くの利用者にとってブラウザから完結できるものにする。
3. 音声パイプラインを観測可能かつデバッグ可能にし、AI Stack-chan を安定化する。
4. STT、LLM、TTS ベンダーをアプリケーション書き換えなしに差し替えられるプラガブルな AI パイプラインを導入する。
5. アセットツールと教育向け入口を整備し、エコシステムを拡張する。
6. Vision、Wake Word、Swarming などの高度機能は、コアプラットフォームが安定した後に進める。

## フェーズ詳細

### Phase 0: アーキテクチャとベースライン

このフェーズでは、アーキテクチャとプラットフォームに残る未確定事項を明示的な設計判断に変えます。

対象範囲:

- 2026 年時点で Stack-chan をどのような存在として位置づけるかを明確化する:
  - ハードウェアキット
  - ファームウェアプラットフォーム
  - 会話型ロボット OS
  - 分散 AI エージェント基盤への土台
- まず標準化すべき対象を決める:
  - mod 形式
  - AI パイプライン契約
  - 構造化ログ形式
  - event bus の境界
- 主要成果物に必要なビルドパイプラインとリリースフローを整備する:
  - 変更ごとに何をビルドし、何を検証するかを定義する
  - ファームウェア、mod、Web ツールのリリース境界を定義する
  - リリースを再現可能かつレビュー可能にする
- AI駆動開発の基盤を定義する:
  - リポジトリ単位の `Skills` と `AGENTS.md` の規約を定義する
  - AI エージェントが自走できる作業範囲と、人手レビューが必要な境界を定義する
  - 自走変更の最小検証ゲートとして `xst` と TypeScript の型チェックを組み込む
- ベースライン指標を採取する:
  - 初回フラッシュまでの時間
  - 初回 MOD 反復までの時間
  - 環境構築の失敗ポイント
  - 参照条件下での TTS 途切れ頻度

推奨成果物:

- `docs/architecture/2026-platform-direction.md`
- `docs/specs/mod-format.md`
- `docs/specs/ai-pipeline.md`
- `docs/specs/log-schema.md`
- `docs/operations/build-pipeline.md`
- `docs/operations/release-flow.md`
- `docs/operations/ai-driven-development.md`
- `AGENTS.md`

### Phase 1: Browser-first 開発体験

このフェーズは、現在の開発体験で特に負担が大きいセットアップの複雑さに対処します。

対象範囲:

- ファームウェア書き込み、設定変更、mod 開発がひとつながりのブラウザ体験になるよう統合する。
- mod をソース主体で扱うか、パッケージ化された bundle にするか、あるいは両対応とした上で正規の転送形式を持つかを定義する。
- ブラウザ主導および LLM 生成 mod に対するサンドボックスと権限制御モデルを設計する。
- ブラウザ実行結果を、人間、CI、LLM のいずれでも再利用できるシナリオ/テスト形式として定義する。
- 現行ランタイムのどこまでを WebAssembly に移せるかを判断するため、現実的なシミュレータ検証を行う。

優先成果:

- 一般的な作業に対する zero-install または minimal-install フロー
- 検証後の one-click device write
- `error_code`、trace、timing、memory を含むログ
- 自動修正ループの停止条件と、フラッシュ前の人手承認方針の明確化

### Phase 2: AI 安定化

このフェーズでは、AI Stack-chan を「有望なデモ」から「診断可能なシステム」へ進めます。

対象範囲:

- 音声パスをエンドツーエンドで計測する。
- TTS 途切れの主要因を切り分ける:
  - ネットワーク遅延
  - バッファアンダーフロー
  - タスクスケジューリング競合
  - GC pause
- 必要に応じてバッファ管理を再設計する。
- AI セットアップを安全性を保ちながらも開発者に過度な負担をかけない秘密情報設定フローに改善する。

優先成果:

- 再現可能な診断シナリオ
- AI 障害に対する共通テレメトリ語彙
- サポート対象デバイスに対する性能上限の文書化

### Phase 3: プラガブル AI パイプライン

このフェーズでは、コスト固定化を避け、実験しやすさを高めます。

対象範囲:

- STT、LLM、TTS 各段階の間のインターフェースを正規化する。
- 主要プロバイダ向けアダプタ実装と、ローカル/クラウド混成の参照構成を提供する。
- 各段階の遅延予算とフォールバック挙動を定義する。
- 可能な限り、アプリケーションロジックがプロバイダ非依存になるよう設計する。

優先成果:

- ベンダー差し替えが可能な参照パイプライン
- アダプタ契約と期待能力に関するドキュメント
- 将来の組み込み会話ロボット統合に向けたベースライン

### Phase 4: エコシステムツーリング

このフェーズでは、プラットフォーム契約が安定した後にクリエイター向けツール群を拡張します。

対象範囲:

- Web ベースの顔アセット制作フローを提供する。
- アセットのパッケージングと転送方式を標準化する。
- 実行時ロードとビルド時埋め込みをどの条件で使い分けるべきかを決める。
- 子どもや初学者向けの Visual Programming レイヤを試作する。
- 上級者向け mod 開発と分断された別系統ではなく、共存可能な形で検証する。

優先成果:

- プラットフォームの正式な成果物を生成できるアセットツール
- 入口を下げつつ、基盤モデルを隠しすぎない educational profile

### Phase 5: 高度機能

これらの項目は重要ですが、プラットフォームの中核が整うまでロードマップの先頭には置きません。

対象範囲:

- Vision ベースのトリガと認識
- Wake Word 検出
- 複数デバイス間の会話と同期動作
- 実運用を見据えたシナリオと多言語シナリオの検証

必要な前提条件:

- 安定した AI パイプライン契約
- 定義済みの event bus と同期モデル
- BLE と Wi-Fi の信頼できる共存戦略
- 対象ハードウェア上での CPU、メモリ、遅延余力の計測

## 依存関係ルール

このロードマップでは、意図的に次の順序制約を課します。

- Browser ツーリングは、Phase 0 で定義されていない暫定 mod 形式に依存してはならない。
- Browser ツーリングと後続フェーズの実装は、Phase 0 で整備したビルドパイプラインとリリースフローを土台にすべきである。
- AI 駆動開発は、Phase 0 で定義した指示モデルと検証ゲートを迂回してはならない。
- Vision、Wake Word、Swarming は、AI 安定化が完了する前に主投資対象として進めてはならない。
- Visual Programming は、上級者向け mod 開発と同じランタイム契約を対象にすべきである。
- より実運用を見据えたシナリオの検討は、ベンダー柔軟な AI パイプライン設計より先行してはならない。

## 成功指標

このロードマップは、機能リリース数ではなく測定可能な成果で評価すべきです。

- 開発者オンボーディング:
  - サポート対象ブラウザから 10 分以内に初回ファームウェア書き込みが成功する
  - ローカル SDK なしで 15 分以内にサンプル mod の反復ができる
- ランタイム品質:
  - TTS 障害に対する構造化テレメトリが取得できる
  - ネットワーク、バッファ、スケジューリング、GC 起因を区別できる診断シナリオがある
- プラットフォーム拡張性:
  - mod パッケージング、ログ、AI アダプタの契約が文書化されている
  - プロバイダ変更時に、アプリケーションレベルの書き換えを最小化できる
- AI 駆動開発:
  - `Skills`、`AGENTS.md`、自走可能な作業境界についてリポジトリ規約が定義されている
  - 自走変更が `xst` と TypeScript の型チェックで検証される
- エコシステム成長:
  - 顔や簡単な振る舞いのブラウザツールが、同一のプラットフォーム基盤上に構築されている

## コア安定化まで延期する項目

以下は価値がありますが、前提条件を満たすまでは明示的に延期対象とします。

- デバイスランタイムと完全同等な wasm シミュレータ
- 多数デバイス向けの本格的な Swarming
- 実運用や多言語対応を見据えた展開パッケージ
- プロトタイプ段階を超えた大規模教育向けツール展開
